# Implementation Plan: Smart Budget Maintenance

**Branch**: `012-smart-budget-maintenance` | **Date**: 2026-02-25 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/012-smart-budget-maintenance/spec.md`

## Summary

Add budget intelligence to the family assistant that detects when YNAB goals have drifted from actual spending, suggests reality-based updates, allocates bonus income to underfunded priorities, runs monthly health checks, and flags stale/mergeable categories. All new functions extend the existing `src/tools/ynab.py` module. Four new Claude tools are added to `assistant.py`, one new n8n-triggered endpoint to `app.py`, and the existing meeting prep flow is enhanced with a budget health section.

**Language/Version**: Python 3.12 (existing codebase)
**Primary Dependencies**: FastAPI, anthropic SDK (Claude Opus for tool loop), httpx (YNAB API), existing WhatsApp/n8n infrastructure
**Storage**: In-memory computation from YNAB API data; pending suggestions stored in `data/budget_pending_suggestions.json` (same pattern as Amazon/email sync)
**Testing**: Manual E2E via WhatsApp + curl endpoint testing (same pattern as Features 010/011)
**Target Platform**: Docker on NUC (existing deployment)
**Project Type**: Extension of existing web service
**Performance Goals**: Health check response within 30 seconds (3-6 YNAB API calls at ~2s each)
**Constraints**: YNAB API rate limit 200 req/hour; `goal_type` is read-only (cannot create new goals via API); WhatsApp message limit ~4096 chars
**Scale/Scope**: ~54 YNAB categories across 17 groups, 2 users

## Key Technical Decisions

### 1. Extend ynab.py rather than create new module

The existing `ynab.py` already has YNAB API setup (base URL, headers, budget ID), caching (`_get_categories()`), fuzzy matching, and all read/write primitives. New functions compose these directly. A separate module would duplicate setup and require cross-module imports.

### 2. No persistent storage for health reports

Health reports and category profiles are computed on-demand from YNAB API data (3-6 API calls). No need to persist them — the YNAB API is the source of truth. Pending suggestions use the same `data/*.json` pattern as Amazon/email sync for conversation continuity.

### 3. Goal target updates via PATCH /categories/{id}

YNAB API allows updating `goal_target` on categories that already have a goal (`goal_type != null`). For categories with no goal, the bot instructs the user to set a goal type in the YNAB app first. This is a known API limitation — documented in the tool's error response.

### 4. AI-inferred priority tiers at runtime

Category tier classification (essential/savings/discretionary) is done by Claude at allocation time using group names as context. No stored tier mappings — the classification prompt is part of the `allocate_bonus` tool's system instructions, keeping it adaptive to YNAB reorganizations.

### 5. Health score = dollar-weighted alignment percentage

Formula: `max(0, (1 - sum(|actual - goal|) / sum(max(actual, goal))) * 100)`. Dollar-weighting prevents small categories from disproportionately affecting the score. Categories without goals are excluded from the score but reported separately.

### 6. Spiky category handling

Categories with coefficient of variation > 1.0 (standard deviation > mean) are flagged as "spiky" and either use a 6-month lookback or are presented with a note ("highly variable — average may be misleading"). The Maintenance category ($186 / $11,068 / $133) is a known example.

## New Functions (ynab.py)

| Function | Purpose | YNAB API Calls |
|----------|---------|----------------|
| `_fetch_month_data(month)` | Get all categories for a specific month | 1 GET /months/{month} |
| `_build_category_profiles(lookback=3)` | Compute CategoryHealthProfile for all categories | 3-6 GET calls |
| `budget_health_check(lookback=3, threshold=30)` | Full health report with drift, stale, merge detection | 3-6 GETs |
| `apply_goal_suggestion(category, amount, apply_all)` | Update goal_target via PATCH /categories/{id} | 1 PATCH per category |
| `allocate_bonus(amount, description)` | Generate prioritized allocation plan | 1 GET (current month) |
| `approve_allocation(adjustments)` | Execute allocation via update_category_budget() | N PATCHes (one per category) |
| `run_budget_health_check()` | Scheduled orchestrator — compute + send WhatsApp | 3-6 GETs + 1 WhatsApp send |
